public class CodeGenerator
extends org.apache.tools.ant.Task

Using the schemas.json file produced by the SchemaExtractor, this Ant task produces a JAR file for use in your application.   A simple example would be: 

<target name="gen-dko-src">
   <taskdef name="dkogen" classname="org.kered.dko.ant.CodeGenerator"
      classpath="${dko_jar_location}"/>

   <dkogen package="com.mycompany.dkos"
      debug="true"
      javaoutputdir="gensrc/dkos"
      schemas="schemas.json"
      datasource="dbgd1 = com.mycompany.dkos.callbacks.MyDataSourceFactory.get();"
   />
</target>

setTypeMappings

public void setTypeMappings(java.lang.String s)
Path to an optional user-written JSON that controls what default types should map to what custom java types and how to convert those types. For instance, SQL timestamp columns by default map to java.sql.Timestamp classes. If instead you'd prefer java.util.Dates: 
{
     "class_mappings": {
         "java.sql.Timestamp": "java.util.Date",
     },
     "functions": {
       "java.sql.Timestamp java.util.Date" :
           "new java.util.Date((%s).getTime()",
       "java.util.Date java.sql.Timestamp" :
           "new java.sql.Timestamp((%s).getTime()",
     }
 }
Note the "functions" section where you can provide code that converts from one to another.

If you don't want all types to be mapped the same way for all classes, you can specify Java regex statements (matched against the schema/table/column names) to specify certain classes. For example, this will do the same as the previous example, but only for fields with names ending in "_date": 

{
     "schema_mappings": {
         ".*_date": "java.util.Date",
     },
     "functions": {
       "java.sql.Timestamp java.util.Date" :
           "new java.util.Date((%s).getTime()",
       "java.util.Date java.sql.Timestamp" :
           "new java.sql.Timestamp((%s).getTime()",
     }
 }

setFakeFKs

public void setFakeFKs(java.lang.String s)
Usually DKOs use the automatically generated FK relationships extracted from the database. But sometimes those foreign keys aren't actually there. (usually for performance reasons) Specifying this file lets you add FK relationships to the generated code regardless. The format is JSON like the following: 
{
     "fk_1":{
         "reffing": ["my_schema","product"],
         "reffed": ["my_schema","manufacturer"],
         "columns": {"manufacturer_id": "id"}
     },
 }
This will create a FK relationship from product.manufacturer_id to manufacturer.id. Compound keys are simply additional entries in the "columns" map.

setCallbackPackage

public void setCallbackPackage(java.lang.String s)
DKOs support a variety of callbacks pre and post database operations. Specify this parameter for where the generated DKOs should look. For instance, a schema/table myapp.product could generate a class org.kered.myapp.Product. Which if given a callback package of "org.kered.myapp.callbacks" would look for callback methods in org.kered.myapp.callbacks.ProductCB.

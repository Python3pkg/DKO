public interface Query<T extends Table>
extends java.lang.Iterable<T>

The Query interface is the center of the Nosco API.   When you generate your JAR file (with org.kered.dko.ant.CodeGenerator) each object gets its own static Query instance.   For example, if your database table looked like this:

some_class
idname
123my name
456your name

The CodeGenerator would create the following for you (simplified): 

  public class SomeClass extends Table {
     final static Query<SomeClass> ALL = new Query<SomeClass>();
     final static Field<Integer> ID = new Field<Integer>();
     final static Field<String> NAME = new Field<String>();
  }
The constant SomeClass.ALL will generally be your starting point for all queries expected to return SomeClass objects.   For example, if you wanted to iterate over all of them you could do this: 
  for (SomeClass x : SomeClass.ALL)
     System.out.println(x);
If you only want a specific one (assuming "id" was the primary key for the table) you could do this: 
  SomeClass x = SomeClass.ALL.get(SomeClass.ID.eq(123)))
SomeClass.ID is a Field<Integer>. All Field<R> objects contain a eq(<R> x) which returns a Condition object. (here passed into Query.get(Condition... c))   Similarly: 
  for (SomeClass x : SomeClass.ALL.where(SomeClass.NAME.like("%me%")))
     System.out.println(x);
would print out all rows named like "me".

count

long count()
           throws java.sql.SQLException
Returns the database-calculated count of the query. Does not download the objects into the JVM. Much faster than .asList().size() or counting the objects yourself. WARNING: Joins can inflate this number and cause a apparent discrepancy with .asList().size(). For example, if table A has a FK to table B and you do a query like B.ALL.with(A.B_FK) the size counted by the database likely will be different from the size counted by you counting objects. This is because the database does not merge objects into sets for you like DKOs do.
